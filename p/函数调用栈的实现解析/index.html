函数调用栈的实现解析

2026-07-24 17:31
2026-07-24 18:25
阅读时长: 16 分钟
3198 字

函数调用栈（Call Stack）是程序运行时的核心机制之一。每当一个函数被调用，系统会为其分配一块连续的内存区域——称为栈帧（Stack Frame）——用于存储局部变量、参数、返回地址等信息。

本文基于 ARM64 架构（aarch64），所有汇编代码均来自 aarch64-linux-gnu-gcc -O0 -fno-omit-frame-pointer 的真实编译输出。后续另有一篇对照文章介绍 x86-64 的实现。

CPU 执行指令的基本原理

在深入函数调用栈之前，有必要先了解 CPU 执行指令的基本机制。CPU 内部有多种不同功能的寄存器，其中与指令执行直接相关的有三个：

寄存器 全称 作用
PC Program Counter（程序计数器） 存放下一条指令的内存地址。在 ARM64 中也称为 PC
IR Instruction Register（指令寄存器） 存放当前正在执行的指令
SR Status Register（状态寄存器） 存储 CPU 当前的状态，如条件标志位（零标志 ZF、进位标志 CF 等）、中断禁止位、处理器模式标志等。在 ARM64 中称为 NZCV（Negative, Zero, Carry, Overflow）

此外，CPU 还拥有大量用于存储数据和地址的寄存器，根据用途可分为整数寄存器、浮点数寄存器、向量寄存器和地址寄存器等。其中，有些寄存器既可以存放数据，又可以存放地址，被称为通用寄存器（GR，General Register）——ARM64 中的 x0–x30 均属此类，w0–w30 则用于访问它们的低 32 位。

  1. 取指 IF（Instruction Fetch）：CPU 根据 PC 寄存器中的地址，从指令存储器或 Cache 中读取指令到 IR 寄存器。
  2. 译码/读寄存器 ID（Instruction Decode）：操作控制器对指令进行译码，同时从寄存器堆中读取操作数。
  3. 执行/计算地址 EX（Execute）：ALU 执行运算操作或计算内存地址。
  4. 访存 MEM（Memory Access）：对存储器（数据 Cache）进行读写操作。
  5. 写回 WB（Write Back）：将指令执行结果写回寄存器堆。然后 PC 自增，进入下一轮循环。

ARM64 特点：ARM64 是定长 32 位指令编码（所有指令均为 4 字节），因此 PC 每次自增 4。这与 x86 的变长编码不同，反汇编时的地址总是间隔 4——从后面的代码中可以清晰看到。

这个"取指 → 译码/读寄存器 → 执行/计算地址 → 访存 → 写回 → PC 自增"的循环，是整个程序运行的底层根基。而函数调用栈正是在这个循环之上，通过 bl（Branch and Link，修改 PC + 保存返回地址到 x30）和 ret（跳转到 x30）等指令，实现控制流的跳转与回归。接下来的章节，我们将从进程的宏观地址空间开始，逐步深入到每一行汇编指令的微观世界。

关键观察：

  • 栈在高地址端，向低地址增长：每次函数调用（stp / sub sp, sp, #N）使得 SP（栈指针）减小。
  • 堆向高地址增长：malloc/new 分配的内存从低往高走。
  • 栈与堆相向生长，中间是大片空闲空间，除非耗尽才会重叠（栈溢出）。

// func_stack.c
#include <stdio.h>

int Sum(int x, int y)
{
    int s = x + y;
    return s;
}

int Calc(int i, int j)
{
    int sum = Sum(i, j);
    int result = i * j - sum;
    return result;
}

int main()
{
    int a = 3;
    int b = 5;
    int res = Calc(a, b);
    printf("result = %d\n", res);
    return 0;
}

调用链一目了然：main → Calc → Sum。

ARM64（AArch64）ABI 调用约定速览

在继续分析汇编之前，需要了解几个关键约定（ARM Procedure Call Standard）：

规则 说明
参数传递 前 8 个整数参数依次放入 x0–x7（32 位用 w0–w7）；浮点参数用 v0–v7；更多参数压栈
返回值 整数/指针放入 x0（w0）
被调用者保存（Callee-saved） x19–x28——被调用者若使用需自己保存恢复；此外 x29（FP）和 x30（LR）也需在帧中保存
调用者保存（Caller-saved） x0–x18——调用者若需要后续使用须自行保存
帧指针 x29 用作帧指针（FP），指向当前栈帧的基址
链接寄存器 x30 保存 bl 指令设置的返回地址——不压栈，而是存在寄存器中
栈对齐 SP 必须始终 16 字节对齐；ARM64 没有 Red Zone，非叶子函数必须通过调整 SP 分配栈空间
指令长度 所有指令固定 4 字节（32 位），PC 自增 4

与 x86-64 的关键差异

ARM64 的函数调用栈与 x86-64 有一个本质区别：

  • x86-64：call 指令将返回地址压入栈中，ret 从栈中弹出。
  • ARM64：bl（Branch and Link）将返回地址存入寄存器 x30（LR），ret 从 x30 跳回。如果被调函数还要调用其他函数（非叶子函数），则必须手动将 x30 保存到栈上。

这意味着 ARM64 的叶子函数（如本文中的 Sum）甚至不需要碰栈来保存返回地址——返回地址一直安安稳稳地躺在 x30 里。

关键观察：

  • Sum 是叶子函数（不再调用其他函数），无需保存 x29/x30——返回地址安全地存放在 x30 中。
  • 即使加了 -fno-omit-frame-pointer，Sum 作为叶子函数仍未设置 x29 帧指针——所有局部变量通过 SP 相对寻址访问。-fno-omit-frame-pointer 主要确保非叶子函数生成帧指针（如 Calc 和 main），以维护 x29 链用于栈回溯（backtrace）。
  • 32 字节（0x20）的栈分配看起来很大，但这是因为 -O0 为每个变量都分配了独立的位置，并保留了对齐余量。
  • 所有局部变量都通过 sp + 偏移 访问。

  7a8:   b9402be0   ldr     w0, [sp, #40]           ;  w0 = sum
  7ac:   4b000020   sub     w0, w1, w0              ;  w0 = i*j - sum
  7b0:   b9002fe0   str     w0, [sp, #44]           ;  result = w0 → sp+44
  7b4:   b9402fe0   ldr     w0, [sp, #44]           ;  w0 = result (返回值)
  7b8:   a8c37bfd   ldp     x29, x30, [sp], #48     ;  恢复 x29/FP、x30/LR，释放 48 字节
  7bc:   d65f03c0   ret                             ;  返回 (跳转至 x30)

关键指令解读：

  • stp x29, x30, [sp, #-48]! — ARM64 标志性指令。“Store Pair”：将 x29（FP）和 x30（LR）当前的值成对复制到栈内存中。[sp, #-48]! 是前索引（pre-index）寻址：先 sp -= 48，再存。一条指令完成了分配栈空间 + 保存 FP 值 + 保存 LR 值三件事。

    澄清：“保存寄存器"是指把寄存器中的值复制一份到栈上，寄存器硬件本身依然存在且持有原值。之所以要保存，是因为后续指令会覆盖这些寄存器——例如 mov x29, sp 会写入新的帧指针地址覆盖 x29，bl Sum 会把新的返回地址写入 x30 覆盖旧值。如果不提前备份，Calc 返回时就无法恢复调用者（main）的帧指针和返回地址。执行 stp 后栈内存中就有了这两个值的副本，ldp 再将它们读回寄存器完成恢复。

  • mov x29, sp — 建立帧指针，x29 成为当前栈帧的锚点。
  • bl 754 <Sum> — Branch and Link：将返回地址（0x798，即 bl 的下一条指令）写入 x30，然后跳转到 Sum。
  • ldp x29, x30, [sp], #48 — Load Pair 的后索引（post-index）寻址：先加载，再 sp += 48。一条指令完成了恢复 FP + 恢复 LR + 释放栈空间三件事，与 stp 完美对称。
  • 为什么 Calc 用了 48 字节而 Sum 只用 32？因为 Calc 是非叶子函数，需要保存 x29/x30（16 字节）+ 局部变量 sum 和 result + 保存的参数 i/j + 对齐。

位置 内容 谁写入
[sp + 偏移] 高地址 局部变量 str 指令
[sp + 8] 保存的 x30（LR） stp x29, x30, [sp, #-N]!
[sp] = x29 保存的 x29（FP） 同上
x30（寄存器） 返回地址（叶子函数直接使用） bl 指令

ARM64 关键指令语义

指令 作用
sub sp, sp, #N 分配 N 字节栈空间（-O0 叶子函数使用）
add sp, sp, #N 释放 N 字节栈空间
stp x29, x30, [sp, #-N]! 前索引 Store Pair：sp -= N，然后 [sp] = x29，[sp+8] = x30。分配空间 + 保存 FP/LR 一步完成
ldp x29, x30, [sp], #N 后索引 Load Pair：x29 = [sp]，x30 = [sp+8]，然后 sp += N。恢复 FP/LR + 释放空间一步完成
mov x29, sp 建立帧指针，x29 成为栈帧锚点
bl <label> Branch and Link：x30 = 返回地址（下一条指令地址），然后跳转到 <label>
ret 返回：跳转到 x30 中的地址（相当于 br x30）
str w0, [sp, #M] 将 w0（32 位）存入 sp + M
ldr w0, [sp, #M] 从 sp + M 加载 32 位到 w0

栈帧寻址方式差异

x86-64 和 ARM64 除了指令不同，栈帧内部的寻址方式也有一个值得注意的区别：

  • x86-64：典型栈帧以 rbp 为锚点，局部变量在低地址方向（rbp - 4、rbp - 8……），返回地址和参数在高地址方向（rbp + 8、rbp + 16……）。这是因为 call 将返回地址压栈后 RSP 向低地址移动，然后 push rbp 再 mov rbp, rsp，使得 rbp 正好处于返回地址和局部变量之间的"分界线"上。
  • ARM64：局部变量既可以基于 x29（帧指针）寻址（x29 - 4 / x29 + 12），也可以直接基于 SP 寻址（sp + 8 / sp + 28）。本文 -O0 -fno-omit-frame-pointer 下，Calc 和 main 通过 x29 建立了帧指针但实际访问仍用 SP 偏移，Sum 甚至完全省略了帧指针。ARM64 的 stp x29, x30, [sp, #-N]! 一步完成分配空间和保存，不改变 x86 那种"返回地址在帧顶"的内存布局。

栈溢出与保护

虽然栈机制精妙，但它也是攻击者的主要目标：

  • 栈缓冲区溢出（Stack Buffer Overflow）：当局部数组越界写入时，可能覆盖保存的 x29（FP）甚至 x30（LR），劫持控制流——ARM64 上攻击者覆盖 x30 即可重定向 ret 的目标。
  • 防护机制：
    • 栈金丝雀（Stack Canary）：在局部变量和保存的 FP/LR 之间插入一个随机值（从 tpidr_el0 或类似线程本地存储取），返回前检查是否被篡改。
    • ASLR（Address Space Layout Randomization）：随机化栈基址，使攻击者难以预测地址。
    • NX（No-Execute）：栈页标记为不可执行，防止 shellcode 注入。
    • PAC（Pointer Authentication Code）：ARM64v8.3 引入，用密钥对返回地址进行签名，ret 前验证——是 ARM64 特有的强大防护。
    • BTI（Branch Target Identification）：ARM64v8.5 引入，类似 x86 的 CET，限制间接跳转的目标必须为 BTI 指令。

总结

  • 函数栈是程序运行时的"工作台”，每次函数调用创建一个栈帧，bl→ret 构建了一个天然的 LIFO 调用链。
  • ARM64 的核心差异：返回地址存于 x30 寄存器而非栈上——叶子函数免去了栈操作，非叶子函数通过 stp/ldp 成对保存/恢复 x29（FP）和 x30（LR）。
  • x29（FP）是栈帧的"锚点"，但在 ARM64 中并非强制——编译器在 -O0 下设置它便于调试，优化后可能完全省略。
  • SP 始终指向栈顶（最低有效地址），ARM64 要求 SP 保持 16 字节对齐。
  • ARM64 定长指令（4 字节）的设计让指令预取和流水线更简单，配合 x30 链接寄存器，函数调用的开销比 x86 更低。
  • stp/ldp 的成对内存访问是 ARM64 的特色——一条指令完成两笔数据移动，节省代码大小和执行时间。
  • 理解函数栈不仅有助于调试和性能分析，也是系统安全的基础——无论是漏洞利用还是防御，都绕不开对栈的深刻理解。
