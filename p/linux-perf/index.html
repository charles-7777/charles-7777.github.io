概述

perf 全称 performance counters.Performance counters for Linux are a new kernel-based subsystem that provide a framework for all things performance analysis. It covers hardware level (CPU/PMU, Performance Monitoring Unit) features and software features (software counters, tracepoints) as well. 源码位于Linux 源码中：包括用户态工具（tools/perf）和 内核子系统（kernel/events/）,还有arch/arm64/kernel/perf_xxx.c 和driver/perf/

perf 是 Linux 内核性能事件子系统（perf_event）的用户态工具集，利用 PMU（Performance Monitoring Unit）硬件计数器、软件事件和内核 tracepoint 来量化系统行为。与 ptrace 的"停止-读取-继续"模式不同，perf 在内核侧通过环形缓冲区（ring buffer）将采样数据批量上传到用户态，不会阻塞被观测进程。perf 最初由 Thomas Gleixner 和 Ingo Molnar 等人开发，2009 年随 Linux 2.6.31 合入主线。它既是一个计数工具（“发生了多少次”），也是一个采样工具（“在哪里发生的”）。

perf — Performance Event

perf是一个面向event的可观测性工具，能够帮助您实现高级性能分析和故障排查功能。它可以回答的问题包括：

  • 为什么内核在 CPU 上运行时间这么长？具体是哪些代码路径导致的？
  • 哪些代码路径导致了 CPU 二级缓存缺失？
  • CPU 是否因内存 I/O 而停滞？
  • 哪些代码路径在分配内存，分配了多少？
  • 是什么触发了 TCP 重传？
  • 某个内核函数是否被调用，调用频率是多少？
  • 线程离开 CPU 的原因是什么？

perf 的使用依赖我们前面所说的 event(事件)。event 是不同内核工具框架的统一接口，上面的图片说明了 event 来源:

  • Hardware Events: CPU性能监视计数器 PMCs
  • Software Events: 这些是基于内核计数器的低级事件。例如，CPU迁移、主次缺页异常等等。
  • Kernel Tracepoint Events: 硬编码在内核中的静态内核级的检测点，即静态探针
  • User Statically-Defined Tracing (USDT): 这些是用户级程序和应用程序的静态跟踪点。
  • Dynamic Tracing: 可以被放置在任何地方的动态探针。对于内核软件，它使用kprobes框架。对于用户级软件，uprobes。
  • Timed Profiling: 使用perf -FHz选项以指定频率收集的快照。这通常用于CPU使用情况分析，其工作原理是周期性的产生时钟中断事件。

// tools/perf/perf-sys.h
static inline int
sys_perf_event_open(struct perf_event_attr *attr,
		      pid_t pid, int cpu, int group_fd,
		      unsigned long flags)
{
	int fd;

	fd = syscall(__NR_perf_event_open, attr, pid, cpu,
		     group_fd, flags);

#if HAVE_ATTR_TEST
	if (unlikely(test_attr__enabled))
		test_attr__open(attr, pid, cpu, fd, group_fd, flags);
#endif
	return fd;
}

struct perf_event_attr {
    __u32 type;               // PERF_TYPE_HARDWARE / SOFTWARE / TRACEPOINT / ...
    __u32 size;               // sizeof(struct perf_event_attr)
    __u64 config;             // 事件具体配置（如 PERF_COUNT_HW_CPU_CYCLES）
    union {
        __u64 sample_period;  // 采样周期（模式）
        __u64 sample_freq;    // 采样频率（模式）
    };
    __u64 sample_type;        // 采样数据包含哪些字段（IP, TID, TIME, ADDR, ...）
    __u64 read_format;        // 读取格式
    // ... 更多标志位
    __u64 disabled       : 1; // 初始是否禁用
    __u64 inherit        : 1; // 是否跟踪子进程
    __u64 pinned         : 1; // 是否独占 PMU
    __u64 exclusive      : 1;
    __u64 exclude_user   : 1; // 排除用户态
    __u64 exclude_kernel : 1; // 排除内核态
    __u64 exclude_hv     : 1; // 排除虚拟机管理程序
    __u64 precise_ip     : 2; // 精确指令指针（PEBS 精度）instruction pointer(ip)
    // ...
};

  • 指标解读
    指标 (Event) 含义 (Description) 解读与分析 (Interpretation & Analysis)
    task-clock 任务占用的 CPU 时间（毫秒） CPU 利用率：该值高说明程序是 CPU 密集型（CPU-bound），反之则可能受 I/O 或调度等待影响。事件名带 :u 后缀表示仅统计用户态。
    context-switches 上下文切换次数 调度开销：频繁切换会带来性能开销。如果数值过高，可能暗示线程/进程数过多或锁争用严重。
    cpu-migrations CPU 迁移次数 CPU 亲和性：进程被调度器在不同 CPU 核心间迁移的次数。频繁迁移会降低缓存命中率。
    page-faults 缺页异常次数 内存访问：包括次要缺页（内存已映射）和主要缺页（需磁盘 I/O）。主要缺页对性能影响很大。
    cycles CPU 时钟周期数 执行时长：指示程序运行消耗的 CPU 周期。可结合 task-clock 计算 CPU 频率。
    instructions 执行的指令数 指令量：程序执行的机器指令总数。是衡量代码复杂度的指标之一。
    branches 分支指令数 分支操作：程序中条件跳转、循环等分支指令的总数。
    branch-misses 分支预测失败次数 预测惩罚：分支预测失败会导致流水线清空。该值比例（如 4.96% of all branches）越低越好。
指标 含义 详细说明
seconds time elapsed 程序总运行时间 (墙上时钟时间) 从程序启动到结束的实际耗时，包含 CPU 计算、I/O 等待、睡眠、调度延迟等所有时间。这是用户体验到的“真实时间”。
seconds user 用户态 CPU 时间 程序在用户空间执行代码所消耗的 CPU 时间总和（多核累加）。反映应用程序自身逻辑的计算开销。
seconds sys 内核态 CPU 时间 程序在内核空间执行代码所消耗的 CPU 时间总和（多核累加）。反映系统调用、文件 I/O、内存分配、网络收发等内核服务的开销。

void __init perf_event_init(void)
{
    int ret;

    idr_init(&pmu_idr);                                    // PMU ID 分配器
    perf_event_init_all_cpus();                             // 初始化所有 CPU 的 per-CPU 数据
    init_srcu_struct(&pmus_srcu);                           // PMU 的 SRCU 保护
    perf_pmu_register(&perf_swevent, "software", PERF_TYPE_SOFTWARE);  // 软件 PMU
    perf_pmu_register(&perf_cpu_clock, NULL, -1);           // CPU 时钟 PMU
    perf_pmu_register(&perf_task_clock, NULL, -1);          // 任务时钟 PMU
    perf_tp_register();                                     // tracepoint PMU
    perf_event_init_cpu(smp_processor_id());                // 初始化引导 CPU
    register_reboot_notifier(&perf_reboot_notifier);        // 重启通知链

    ret = init_hw_breakpoint();                             // 硬件断点初始化
    WARN(ret, "hw_breakpoint initialization failed with: %d", ret);

    /* 编译时验证 data_head 偏移量 == 1024，这是用户空间 ABI 的一部分 */
    BUILD_BUG_ON((offsetof(struct perf_event_mmap_page, data_head)) != 1024);
}

__init int init_hw_perf_events(void)
{
    int err;

    pr_info("Performance Events: ");
    // 根据 CPU 厂商分发
    switch (boot_cpu_data.x86_vendor) {
    case X86_VENDOR_INTEL:
        err = intel_pmu_init();         // Intel PMU 初始化
        break;
    case X86_VENDOR_AMD:
        err = amd_pmu_init();           // AMD PMU 初始化
        break;
    default:
        return 0;
    }

    if (err != 0)
        return err;

    // 设置 NMI 中断
    perf_events_lapic_init();           // 写 APIC LVTPC 寄存器

    // 注册 NMI handler
    register_nmi_handler(NMI_LOCAL,
                         perf_event_nmi_handler,
                         0, "PMI");
    return 0;
}

static int perf_event_nmi_handler(unsigned int cmd, struct pt_regs *regs)
{
    u64 start_clock;
    u64 finish_clock;
    int ret;

    // 当前没有任何 active events 时跳过（可能被其他 NMI 源触发）
    if (!atomic_read(&active_events))
        return NMI_DONE;

    start_clock = sched_clock();
    // 通过 x86_pmu 函数指针分发到具体实现
    ret = x86_pmu.handle_irq(regs);
    finish_clock = sched_clock();

    // 统计处理耗时（用于动态调整采样率控制）
    perf_sample_event_took(finish_clock - start_clock);
    return ret;
}

// arch/x86/events/intel/core.c:2522-2592
static int intel_pmu_handle_irq(struct pt_regs *regs)
{
    struct cpu_hw_events *cpuc = this_cpu_ptr(&cpu_hw_events);
    u64 status;
    int handled = 0;

    // 1. 保存 PMU 状态
    cpuc->enabled = 0;

    // 2. 禁用 PMU
    __intel_pmu_disable_all();

    // 3. 处理 BTS（Branch Trace Store）
    intel_pmu_drain_bts_buffer();
    intel_bts_interrupt();

    // 4. 读取溢出状态寄存器
    status = intel_pmu_get_status();        // 读 MSR_GLOBAL_STATUS
    if (!status)
        goto done;

again:
    // 5. 保存 LBR 记录
    intel_pmu_lbr_read();

    // 6. 确认中断（清除状态位）
    intel_pmu_ack_status(status);

    // 7. 核心处理
    handled += handle_pmi_common(regs, status);

    // 8. 检查是否有新的中断（NMI 重入）
    status = intel_pmu_get_status();
    if (status) {
        if (++loops > 100) {   // 防止无限循环
            intel_pmu_handle_irq_reset(regs);
            goto done;
        }
        goto again;
    }

done:
    // 9. 恢复 PMU
    __intel_pmu_enable_all();
    return handled;
}

// arch/x86/events/intel/core.c:2332-2427
static int handle_pmi_common(struct pt_regs *regs, u64 status)
{
    struct cpu_hw_events *cpuc = this_cpu_ptr(&cpu_hw_events);
    int bit;
    struct perf_event *event;

    // 统计中断
    inc_irq_stat(apic_perf_irqs);

    // 清除不需要处理的位
    status &= ~(GLOBAL_STATUS_COND_CHG | GLOBAL_STATUS_ASIF |
                GLOBAL_STATUS_LBRS_FROZEN);

    // PEBS 溢出处理（bit 62）
    if (status & GLOBAL_STATUS_PEBS_BIT) {
        drain_pebs(regs, cpuc->ds);
        status &= ~GLOBAL_STATUS_PEBS_BIT;
    }

    // Intel PT 处理（bit 55）
    if (status & GLOBAL_STATUS_TRACE_TOPAPMI_BIT) {
        intel_pt_interrupt();
        status &= ~GLOBAL_STATUS_TRACE_TOPAPMI_BIT;
    }

    // 遍历每个溢出事件
    for_each_set_bit(bit, (unsigned long *)&status, x86_pmu.num_counters) {
        event = cpuc->events[bit];
        if (!test_bit(bit, cpuc->active_mask))
            continue;

        // 保存当前计数值并重启计数器
        intel_pmu_save_and_restart(event);

        // 通知 perf 子系统事件溢出
        perf_sample_data_init(&data, 0, event->hw.last_period);
        if (perf_event_overflow(event, &data, regs))
            x86_pmu_stop(event, 0);  // 如果溢出处理失败，停止事件
    }

    return handled;
}

static int __perf_output_begin(struct perf_output_handle *handle,
                               struct perf_event *event, unsigned int size)
{
    struct ring_buffer *rb;
    unsigned long tail, offset, head;
    int have_lost;
    struct perf_sample_data sample_data;

    rcu_read_lock();

    // 1. 获取环形缓冲区
    rb = READ_ONCE(event->rb);

    // 2. 检查缓冲区状态
    if (rb->paused || rb->lost) {
        // 缓冲区暂停或数据已丢失
    }

    // 3. 原子推进写位置（CAS 循环，处理 NMI 重入）
    do {
        tail = READ_ONCE(rb->user_page->data_tail);    // 读取消费进度
        offset = head = local_read(&rb->head);
        head += size;                       // 前进 size 字节

        // 检查是否 wrap 到下一页面
    } while (local_cmpxchg(&rb->head, offset, head) != offset);

    // 4. 计算写入地址
    handle->addr = (void *)((unsigned long)rb->data_pages[offset >> PAGE_SHIFT] +
                            offset % PAGE_SIZE);
    handle->size = size;

exit:
    rcu_read_unlock();
    return 0;
}

// kernel/events/core.c:3555-3588
static void perf_event_context_sched_in(struct perf_event_context *ctx,
                                        struct task_struct *task)
{
    struct perf_cpu_context *cpuctx;

    cpuctx = __get_cpu_context(ctx);

    // 如果已经是当前 task_ctx，跳过
    if (cpuctx->task_ctx == ctx)
        return;

    perf_ctx_lock(cpuctx, ctx);

    // 没有事件时跳过
    if (!ctx->nr_events)
        goto unlock;

    perf_pmu_disable(ctx->pmu);

    /*
     * 优先级排序：
     *   CPU pinned（不需要移动）> 任务 pinned >
     *   CPU flexible > 任务 flexible
     *
     * 如果任务的 context 没有 pinned 事件，
     * 则不需要翻转 cpuctx 的事件
     */
    if (!RB_EMPTY_ROOT(&ctx->pinned_groups.tree))
        cpu_ctx_sched_out(cpuctx, EVENT_FLEXIBLE);

    // 调度事件到 PMU
    perf_event_sched_in(cpuctx, ctx, task);

    perf_pmu_enable(ctx->pmu);

unlock:
    perf_ctx_unlock(cpuctx, ctx);
}

// kernel/events/core.c:3742-3807（简化）
static void perf_adjust_freq_unthr_context(struct perf_event_context *ctx,
                                           int needs_unthr)
{
    list_for_each_entry_rcu(event, &ctx->event_list, event_entry) {
        if (event->state != PERF_EVENT_STATE_ACTIVE)
            continue;

        hwc = &event->hw;

        // 【取消节流】硬件中断过多被暂停，现在恢复
        if (hwc->interrupts == MAX_INTERRUPTS) {
            hwc->interrupts = 0;
            perf_log_throttle(event, 1);
            event->pmu->start(event, 0);       // 重新启动
        }

        // 【频率调整】动态调整采样周期
        if (!event->attr.freq || !event->attr.sample_freq)
            goto next;

        event->pmu->stop(event, PERF_EF_UPDATE);
        now = local64_read(&event->count);
        delta = now - hwc->freq_count_stamp;
        hwc->freq_count_stamp = now;

        if (delta > 0)
            perf_adjust_period(event, period, delta, false);

        event->pmu->start(event, delta > 0 ? PERF_EF_RELOAD : 0);
    }
}

{

    int ret = 1;

    // 累计中断次数

__perf_event_output(struct perf_event *event,
                    struct perf_sample_data *data,
                    struct pt_regs *regs,

10.3 核心工作机制总结

方面 机制 说明
计数模式 pmu->read() 用户通过 read(fd) 或 ioctl(fd, PERF_EVENT_IOC_READ) 读取 event->count
采样模式 Ring Buffer PMI 触发 → 采样 → 写入 ring buffer → 用户通过 mmap 读取
频率控制 自适应周期 Timer tick 中根据实际采样率动态调整 sample_period
节流保护 MAX_INTERRUPTS 当事件中断太频繁（中断次数超阈值）时暂停事件，tick 中恢复
无锁同步 smp_wmb() + seq-lock 内核写 data_head、用户读 data_head + 写 data_tail，无锁
NMI 安全 nest 计数 + CAS NMI 内可以重入，通过 nest 计数保证只有最外层提交
调度亲和 sched_in/sched_out 进程切换时 perf 事件上下文随任务迁移，优先级 pinned > flexible
权限控制 perf_event_paranoid 三级安全检查（tracepoint/cpu/kernel）
PMU 抽象 struct pmu + 函数指针 每种 PMU 硬件/软件按统一接口注册，perf 核心层透明调度
事件分组 group_leader + group_node 多个事件可以组成一个 group，原子性地调度到 PMU 上

写在最后：Linux perf 子系统是一个从 start_kernel 最早初始化的子系统之一（开中断之前），到 arch 层的 NMI 中断处理，再到调度器的每次进程切换，再到用户空间的 mmap 无锁读取——贯穿了整个内核生命周期。它的设计哲学是将计数和采样两个核心抽象为统一的 perf_event 对象，通过 PMU 抽象层隔离硬件差异，通过 ring buffer 实现内核到用户的零拷贝数据传输，堪称 Linux 内核中可观测性设施的基石。
